01 — Overview

系統總覽

顧客線上選分區、留問題（問卦），小老師在後台逐筆解讀回覆；顧客用 Google／Email 登入，回覆會主動通知。整個 app 是單一 Python 檔，資料放 Supabase。

正式網址

iching-insight.streamlit.app

  • ⚠ 不是 repo 名 heng-yi-kunze
  • 所有 URL（email 連結／Google redirect／LIFF）都要用這個

技術棧

Streamlit（Python）＋ Supabase（PostgREST）

  • 前端＋後端都在 app.py 一檔
  • DB 走 requests 直打 PostgREST，不用 supabase-py
  • 託管：Streamlit Community Cloud

原始碼

GitHub zchen114-vit/heng-yi-kunze（public）

  • master 分支自動觸發 Streamlit 重部署
  • 密鑰全在 Secrets，不在 repo

02 — Architecture

架構與一筆問卦的流程

顧客必須先登入才能問卦（每筆問卦綁定身分，日後一定查得到）。問卦寫入 DB 後，把一支猜不到的隨機 token 存進這台裝置的 localStorage 並導向 ?token=…，之後回來自動帶出該筆問卦。

flowchart TD
    A["顧客開站
iching-insight.streamlit.app"] --> B{已登入?}
    B -- 否 --> L["Google / Email / LINE 登入"]
    L --> B
    B -- 是 --> C["選分區 → 填姓名+問題"]
    C --> D["create_session()
寫 sessions 列 + 隨機 token"]
    D --> E["add_message()
寫第一則 customer 訊息"]
    E --> F["localStorage 存 token
導向 ?token=XXX"]
    F --> G["顧客進聊天室等回覆
(每 20 秒自動刷新)"]
    D -. LINE 推播 .-> H["小老師收通知"]
    H --> I["後台逐筆解讀回覆
add_message consultant"]
    I -. email/LINE .-> J["顧客收回覆通知
點連結用 ?token= 回登"]
關鍵設計：身分只認 token。網址 ?token= 是唯一安全的「回來查看」入口；舊版的 ?email=／?sid=／?line_uid= 已移除——那些可被任何人手打網址偽造成別人身分（曾是帳號接管漏洞）。token 由 secrets.token_urlsafe(32) 產生，猜不到。
03 — Data Model

資料表（Supabase）

四張表。sessions＝一筆問卦、messages＝對話訊息、config＝管理密碼、settings＝公告與通知錯誤記錄。

表主要欄位用途
sessionssession_id(8碼)、customer_name、category、token、email、line_uid、is_closed(預設 false)、created_at、updated_at、preference(廢欄留空)一筆問卦。is_closed=false＝進行中、後台會撈；true＝歸檔。
messagessession_id、role(customer／consultant)、content、created_at對話訊息。role 決定氣泡是顧客還是小老師。
configkey、value已停用原本存 admin_password；後台 2026-06-26 改 Google 白名單後此表不再使用（可留著不影響）。
settingskey、valueannouncement(跑馬燈)、last_notify_error(最近通知失敗原因)。
欄位是逐步加上去的，靠 migration。repo supabase/migrations/ 下：20260623_add_email_and_line_uid.sql（加 email／line_uid）、20260624_add_session_token.sql（加 token＋index，已跑 2026-06-25）。新增欄位時務必同步寫一支 migration 並在 Supabase SQL Editor 跑過。
04 — Auth

登入與身分

三種登入並存，都接到同一套 email／line_uid 身分機制；登入後若已有進行中問卦會自動帶回。

Google 登入 通

Streamlit 原生 OIDC（st.login／st.user）

  • 需 Secrets [auth] 區塊 + httpx 套件
  • redirect_uri 兩邊（Google Console＋Secrets）必須一字不差

Email 登入

寄 6 位驗證碼（10 分有效、5 次上限）

  • 用 Gmail SMTP 寄信（email_from＋gmail_app_password）
  • 驗證通過 → 記住 email、找進行中問卦

LINE / LIFF

LINE 內開啟免密碼自動登入

  • 需 liff_id＋圖文選單設定
  • 目前次要，Google／Email 為主

token 自動回登

裝置 localStorage 存 iching_token

  • 下次帶 ?token= 自動進該筆問卦
  • 結案／刪除後 token 失效
05 — Keys & RLS

金鑰與 RLS ⚠ 最容易踩雷

app 用 supabase_key 直打 PostgREST。這把金鑰是 service_role（secret） 還是 publishable（anon），決定 RLS 管不管它——這是 2026-06-25 那場「寫得進、讀不回」的根因。

金鑰新版名稱受 RLS 管?該放哪
service_rolesb_secret_…（Secret key）否，繞過 RLS✅ 本系統用這把（只放 Streamlit Secrets）
anonsb_publishable_…是，遵守 RLS給瀏覽器端用；本系統不該用它當 supabase_key
2026-06-25 故障紀錄（顧客「找不到此諮詢記錄」＋後台點不進）— 兩個原因疊在一起：
① 金鑰：supabase_key 一度放成 publishable（受 RLS 管）→ 已換成 secret／service_role（sb_secret_…，繞過 RLS）。已修
② 真兇＝_eqv() 雙引號：防注入用的 _eqv() 把查詢值包成 eq."值"，但這套 PostgREST 不會剝掉雙引號 → 拿含引號的 "值" 比對 → 單筆等值查詢永遠對不到。列表查詢沒走 _eqv（用不加引號的 is_closed=eq.false）所以讀得到——正好造成「列表看得到、用 ID 撈單筆就找不到」。已修（改成白名單清洗＋不加引號，commit 27ab225）。
⚠ 別把 _eqv 改回 eq."值"——這套 PostgREST 不剝引號，會讓所有單筆查詢全部讀不到。
判別工具：?debug=1 →「🔧 DB 自我測試」會印金鑰類型＋「加引號 vs 不加引號」兩種讀回，一翻兩瞪眼。
為什麼用 service_role 是對的？這把金鑰只待在 Streamlit 伺服器、永不送進瀏覽器，所有身分驗證（Google／驗證碼／token）都在伺服器端做完才碰 DB。這就是「可信後端」該有的樣子。前提：絕不可把它寫進程式碼或前端 JS。
06 — Notifications

通知機制

事件通知誰管道
顧客新問卦／追問小老師LINE 推播（line_token＋line_user_id）；失敗原因寫進 settings.last_notify_error，後台看得到
小老師回覆顧客Email（有 email 才寄，附 ?token= 回登連結）＋ LINE 推播（有 line_uid 才推）
寄信走 Gmail SMTP（不是第三方 API），用 email_from 這個 Gmail 帳號 + 16 碼應用程式密碼 gmail_app_password。自家 Gmail 第一封常被分到垃圾匣，驗收時記得翻。
07 — Admin

後台（小老師專用）

從側邊欄「🔐 管理入口」用 Google 帳號登入進入（不再用密碼）。可看進行中問卦清單、逐筆回覆、結案、歸檔、設公告、測 LINE 通知。

  • 授權＝Google 信箱白名單 2026-06-26 改：登入的 Google email 要在 Secrets admin_emails（逗號分隔可多人）裡，才進得了後台。沒有密碼可被暴力破解，MFA／帳號安全／登入限流全交給 Google。要增減管理員 → 改 admin_emails。舊的「密碼登入＋算術驗證碼＋改密碼面板」都已移除。
  • 清單刷新：儀表板每 30 秒自動重抓＋有「🔄 重新整理」鈕（2026-06-25 加，修「切過去看停在舊清單」）。
  • 狀態：🔴 待解讀（最後一則是顧客或還沒提問）／✅ 已解讀（最後一則是小老師）。
  • 篩選：分區、狀態、姓名搜尋、姓氏分組。預設「全部／全部」。
右下角的「Manage app」按鈕＝Streamlit Cloud 平台的，不是本 app 的功能，也跟後台 admin 登入無關。 它由 Streamlit Cloud 帳號把關——只有「登入 share.streamlit.io 且是本 app 擁有者」的你點得動（看 log／Secrets／重啟）。 一般訪客就算看到、用 F12 叫出來也管不了、看不到 Secrets，會被平台登入擋住。 你登出 app 後台還看得到它，是因為瀏覽器仍登著 Streamlit Cloud（屋主身分）＝預期行為，不是漏洞；想看「別人看到的樣子」開無痕視窗即可。 app 第 794–802 行有 CSS 對非 admin 把它藏起來，但那只是化妝（懂的人 F12 能叫回來），真正的牆是平台登入。

09 — How to Change

怎麼改規格（常見修改點）

想改什麼動哪裡
新增／改分區CATEGORIES（app.py 第 54 行）：加一筆，含 icon／desc／welcome。前台卡片、後台篩選自動跟著變。
加 sessions 欄位① 寫 migration 放 supabase/migrations/ 並在 Supabase 跑 ② create_session() 加進 payload ③ 後台顯示處取用。
改回覆通知內容email：notify_customer_reply_email()；LINE：notify_customer_reply_line()。
改後台清單／篩選show_admin()（清單卡片、刷新、篩選）＋ get_all_sessions()（查詢條件）。
改首頁文案／登入區show_home()。（診斷面板已移到後台 show_admin()，見下。）
查線上 DB 狀態先登入後台，再於網址加 ?debug=1 → 儀表板出現「🔧 系統診斷」（金鑰角色＋讀寫往返）。2026-06-26 改為僅 admin 可見（原本任何人加參數就觸發＝洩漏）。
改完上線流程：本機 python -m py_compile app.py 過 → git commit + git push origin master → Streamlit 自動重部署（1–2 分鐘）→ 開 ?debug=1 確認新程式已上線（看得到診斷行＝新版）。
